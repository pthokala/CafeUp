## 1. Product overview

CafeUp is a macOS menu-bar utility that prevents the Mac from going to sleep on user-idle. It exposes three categories of behavior through a single status-bar item:

| Category | Behaviors |
|---|---|
| **Manual sessions** | Indefinite, timed (minutes/hours presets, custom duration, end-at-time) |
| **Conditional sessions** | While a chosen app runs, while downloads are in progress |
| **Triggered sessions** | Automatic activation on app running, schedule, AC power, battery threshold |

The app is a single binary; there is no companion daemon. Everything runs in the main app process, scoped to the logged-in user. All state lives in `UserDefaults` (`com.pardhu.CafeUp.*` keys).

**Bundle identifier:** `com.pardhu.CafeUp`
**Bundle display name:** `CafeUp`
**Version:** `0.2.0 (2)` (`MARKETING_VERSION` / `CURRENT_PROJECT_VERSION` in `project.yml`)

## 4. Active-session panel specification

`ActiveSessionPanel` is a SwiftUI view embedded via `NSHostingView` at the top of the active menu. Fixed width 330pt, dynamic height. Visual goal: match Amphetamine's active-session block.

### Vertical layout (top to bottom)

| Element | Properties |
|---|---|
| Header `Current Session Details:` | `.system(size: 13, weight: .bold)`, foreground `.primary` |
| Status line (`LiveStatusText`) | `.system(size: 13)`, primary; reads `viewModel.tick` so it re-renders each second |
| Activation label | `.system(size: 13)`, `.secondary` |
| Hairline #1 | `Color.primary.opacity(0.08)`, height 0.5pt |
| Toggle row × 3 (`CheckboxRow`) | 13pt label + custom 14×14 square checkbox, 3pt inter-row spacing |
| Hairline #2 | matches #1 |
| Pill button (`endButton`) | 8pt vertical padding above & below; 10pt horizontal indent from panel edges |
| Hairline #3 (panel's natural bottom) | matches above |

Outer panel paddings: 16pt horizontal (matches NSMenu item inset so toggle labels align vertically with `Indefinitely`, `Minutes`, etc. below), 8pt top, 0pt bottom (the closing hairline is the panel's visual edge).

### Status line text rules

`MenuBarViewModel.sessionStatusLine(now:)`:
- No active session → `nil` (the panel section is hidden).
- Indefinite session (`session.endsAt == nil`) → `"Indeterminate time remaining"`.
- Timed session → `"\(amphetamineStyle(secondsRemaining)) (\(clockTime(endsAt)))"`, e.g. `"09m 51s remaining (3:24 PM)"` or `"02h 59m 30s remaining (4:21 PM)"`.
- `RemainingFormatter.amphetamineStyle` uses `HHh MMm SSs remaining` if hours > 0 else `MMm SSs remaining`.

### Activation label

| Manual session | Trigger active | Label |
|---|---|---|
| ✓ | — | `Manual Activation` |
| ✓ | ✓ | `Manual + Triggered Activation` |
| — | ✓ | (shown in trigger-only section instead) |

### Live ticker

`MenuBarViewModel` exposes a `private(set) var tick: Int` that increments every `tickInterval` (default 1 second) while a session is active. `LiveStatusText` reads `tick` so SwiftUI re-evaluates only that subview each second — the wider `MenuBarView` body does not re-run, which would reset NSMenu hover tracking mid-cursor.

Ticker lifecycle:
- Started on every successful `start*` method when `engine.isActive`.
- Re-scheduled inside its own fire closure (each tick schedules the next).
- Stopped (handle cancelled) on `stop()` or when the engine becomes inactive.

### Three wake-behavior toggles

Each `CheckboxRow` is a `Button { isOn.toggle() }` with a custom 14×14 rounded-rect checkbox (3.5pt corner radius, 1pt outline when off, `Color.accentColor` fill + 9pt bold white checkmark when on). Hovering tints the row background `Color.primary.opacity(0.08)`. Bindings:

| Label | Bound property |
|---|---|
| Allow display sleep | `viewModel.policy.allowDisplaySleep` |
| Allow system sleep when display is closed | `viewModel.policy.allowSystemSleepWhenLidClosed` |
| Allow screen saver after 45m of inactivity | `viewModel.policy.allowScreenSaverAfter45Min` |

Setter assigns directly to `viewModel.policy.<field>`; the `policy.didSet` observer:
1. If engine is active, calls `engine.updatePolicy(effectivePolicy)`.
2. Calls `refreshIdleObserver()` to start/stop the idle observer based on the screen-saver toggle.

### End Current Session button

ZStack-composed pill:
- **Center group**: `Image("stop.circle.fill")` (14pt) + `Text("End Current Session")` (13pt, semibold)
- **Right-aligned overlay**: `Text("⌘X")` (12pt, secondary)
- Background: `RoundedRectangle(cornerRadius: 7)`, `Color.primary.opacity(0.10)` fill, no border
- Padding: 14pt horizontal, 7pt vertical inside the pill; 10pt horizontal margin from panel edges; 8pt above and below (between the hairlines)
- ⌘X keyboard shortcut wired via `Button.keyboardShortcut("x", modifiers: .command)`

## 5. Session model & lifecycle

### `Session` value type

```swift
struct Session: Sendable, Equatable, Identifiable {
    let id: UUID
    let mode: SessionMode      // .indefinite | .timed(Duration)
    let policy: WakePolicy
    let startedAt: Date
    let endsAt: Date?          // nil for indefinite
}
```

### `SessionEngine` invariants

`SessionEngine` (a `@MainActor @Observable` class) owns at most one active `Session`. Public API:

| Method | Behavior |
|---|---|
| `start(mode:policy:) throws` | Replaces any existing session. Acquires a fresh `PowerAssertionToken`. Schedules auto-stop via `Scheduler` for timed sessions. Logs at `info`. |
| `stop()` | Releases assertion, cancels scheduled work, sets `current = nil`. No-op when already idle. |
| `updatePolicy(_:) throws` | Re-acquires a token under the new policy *without* changing `startedAt` or `endsAt`. Re-throws if reacquire fails — keeping the original token alive. |

### Failure semantics

- **Initial acquire fails** → throws `SessionError.assertionFailed(code:)`, no session set.
- **Reacquire fails during `updatePolicy`** → throws, but old token + session preserved (the call site sees the error in `lastError`).
- **`updatePolicy` on idle engine** → no-op, no throw.

### Auto-stop scheduling

When `start(mode: .timed(duration))` is called:
- `endDate = startedAt + duration` is computed via `Date.addingTimeInterval`.
- `Scheduler.schedule(after: remaining)` fires a closure that calls `self.stop()` on the main actor.
- The returned `ScheduledWork` handle is stored; `stop()` cancels it.

`TaskScheduler` (the production implementation) wraps `Task.sleep(for:)`. `FakeScheduler` (tests) records all pending work and exposes `fireAll()` for deterministic time advancement.

## 6. Wake policy model & IOKit assertion mapping

### `WakePolicy` struct

```swift
struct WakePolicy: Sendable, Hashable, Codable {
    var allowDisplaySleep: Bool                  // default: false
    var allowSystemSleepWhenLidClosed: Bool      // default: true
    var allowScreenSaverAfter45Min: Bool         // default: false
}
```

Defaults match Amphetamine's out-of-the-box behavior visible in the original app's screenshot.

### Legacy compatibility constants

```swift
WakePolicy.systemOnly       = WakePolicy(allowDisplaySleep: true,  ...)  // legacy enum
WakePolicy.systemAndDisplay = WakePolicy(allowDisplaySleep: false, ...)  // legacy enum
WakePolicy.default          = WakePolicy()
```

### IOKit assertion set

`IOKitPowerAssertionService.acquire(policy:reason:)` holds a *set* of `IOPMAssertion` IDs. Mapping table:

| Condition | Assertion type | Behavior when held |
|---|---|---|
| **Always** (session active) | `kIOPMAssertionTypePreventUserIdleSystemSleep` | System will not sleep due to user idle |
| `!policy.allowDisplaySleep` | `kIOPMAssertionTypePreventUserIdleDisplaySleep` | Display will not sleep due to user idle |
| `!policy.allowSystemSleepWhenLidClosed` | `kIOPMAssertionTypePreventSystemSleep` | System will not sleep even when lid is closed (clamshell mode) |

`policy.allowScreenSaverAfter45Min` does NOT map to a permanent assertion — see [§ 8](#8-idle-aware-screen-saver-release).

### Token lifecycle

`PowerAssertionToken` is a protocol with a single `release()` method.

`IOKitToken` (the production token):
- Stores `[IOPMAssertionID]` for every type held.
- `release()` is idempotent (guarded by an `OSAllocatedUnfairLock`) and calls `IOPMAssertionRelease` for each ID.
- `deinit` calls `release()` as a safety net.

If acquisition of any assertion in the set fails, all previously-acquired IDs in that call are released before throwing — never leaks assertions.

### Effective policy

`policy.effective(idleSeconds:)` returns a copy with `allowDisplaySleep = true` if and only if `allowScreenSaverAfter45Min` is set AND `idleSeconds >= 45 * 60`. This is the policy actually passed to `engine.updatePolicy` when the idle observer fires across the threshold.

## 7. Auto-stop behaviors

Three "auto-stoppers" can be armed for a single session. They are mutually exclusive — starting a new session via any method cancels any prior auto-stopper.

| Auto-stopper | Triggered by | Service | Fires `stop()` when |
|---|---|---|---|
| Timed expiry | Any `start(mode: .timed(duration))` | `Scheduler` | Duration elapses |
| App lifetime | `startWhileAppRunning(bundleIdentifier:)` | `AppLifetimeWatcher` | `NSWorkspace.didTerminateApplicationNotification` fires for the bundle id |
| Downloads idle | `startWhileDownloading()` | `DownloadsMonitor` | Polled `~/Downloads` directory contains zero partial-extension files |

### `AppLifetimeWatcher`

Production: `NSWorkspaceAppLifetimeWatcher`. Subscribes to `NSWorkspace.didTerminateApplicationNotification`; filters by bundle id; calls `onTermination` once and stops observing.

Edge case: if the watched app is not actually running when `watch` is called, the watcher is armed but never fires until/unless that app runs and quits. Starting a new session cancels it.

### `DownloadsMonitor`

Production: `FileSystemDownloadsMonitor`. Polls every 5 seconds (`pollInterval` is configurable).

Partial-file extensions watched:
| Extension | Source |
|---|---|
| `.download` | Safari |
| `.crdownload` | Chrome, Edge, Brave |
| `.part` | Firefox |
| `.partial` | Generic |

Match is case-insensitive. Directory defaults to `FileManager.urls(for: .downloadsDirectory)` and falls back to `~/Downloads`. If the directory is unreadable or missing, `partials(in:)` returns `[]` (so the session immediately stops once `start` is called — current behavior; documented as a known limitation).

### `EndAtTime`

Not a continuous auto-stopper — once the user picks an end time, `MenuBarViewModel.startUntil(_:now:)` computes `seconds = endDate - now` (with same-day past-time wrapping to +24h handled in the picker view), then delegates to `start(duration:)`, which uses the normal timed-expiry path.

## 8. Idle-aware screen-saver release

When `policy.allowScreenSaverAfter45Min` is enabled AND a session is active, `MenuBarViewModel` starts a `UserIdleObserver`.

### `UserIdleObserver` contract

```swift
@MainActor
protocol UserIdleObserver: AnyObject {
    var idleSeconds: TimeInterval { get }
    func start(interval: Duration, onTick: @escaping @MainActor (TimeInterval) -> Void)
    func stop()
}
```

Production: `CGEventSourceIdleObserver`. Uses `CGEventSource.secondsSinceLastEventType(.combinedSessionState, eventType: .init(rawValue: ~0)!)` (matches any input event type). Polls at 30-second intervals by default.

### Effective-policy swap

On each tick, the view model:
1. Records the previous `effectivePolicy` (based on prior `lastIdleSeconds`).
2. Updates `lastIdleSeconds` to the new reading.
3. Recomputes `effectivePolicy`.
4. If it differs AND engine is active, calls `engine.updatePolicy(newEffective)`.

Result: crossing the 45-minute idle boundary releases `PreventUserIdleDisplaySleep` so the screen saver can engage. When the user returns (idle drops back below threshold), the assertion is re-acquired transparently — the user's preference (`allowDisplaySleep`) is preserved as the truth source; `effective` only adds the override when actually idle.

### Observer lifecycle

`refreshIdleObserver()` is called from:
- Every `start*` method (after the session is active).
- `policy.didSet` (so toggling the option mid-session arms/disarms appropriately).
- `stop()` (calls `observer.stop()` directly).

When stopped, `lastIdleSeconds` is reset to 0.

## 9. Trigger system

Triggers are independent of manual sessions: if any trigger is satisfied at any time, the trigger engine holds its own power assertion. Manual session + active triggers can coexist.

### `Trigger` model

```swift
struct Trigger: Codable, Identifiable, Equatable, Sendable {
    let id: UUID
    var name: String
    var isEnabled: Bool
    var conditions: [TriggerCondition]
    var policy: WakePolicy        // default .systemAndDisplay
}
```

A trigger is *satisfied* iff `isEnabled == true`, `conditions` is non-empty, and **all** conditions return true for the current `WorldState` (AND semantics).

### `TriggerCondition` cases

| Case | Satisfied when |
|---|---|
| `.appRunning(bundleIdentifier:)` | The bundle id is present in `WorldState.runningAppBundleIds` |
| `.schedule(weekdays:range:)` | Current weekday ∈ `weekdays` AND current local time ∈ `range` (`TimeRange.contains` supports overnight wrap when `start > end`) |
| `.onACPower` | `WorldState.powerSource.isOnACPower == true` |
| `.batteryAtLeast(percent:)` | `WorldState.powerSource.batteryPercentage != nil` AND `>= percent` |

### `WorldState` snapshot

```swift
struct WorldState: Sendable {
    var runningAppBundleIds: Set<String>
    var powerSource: PowerSource    // isOnACPower + optional batteryPercentage
    var currentDate: Date
}
```

The trigger engine maintains a single `WorldState` and re-evaluates all triggers whenever any of these sources changes:
- `AppActivityObserver` — `NSWorkspace.didLaunchApplicationNotification` / `didTerminateApplicationNotification`.
- `ScheduleObserver` — `TimerScheduleObserver` fires every minute (current time-of-day check).
- `PowerObserver` — `IOPSPowerObserver` subscribes to `IOPSNotificationCreateRunLoopSource`.

### Strictest-policy combination

When multiple triggers are satisfied, the engine acquires assertions for the *strictest* policy:

```swift
// From TriggerEngine.strictestPolicy
return triggers.contains { $0.policy == .systemAndDisplay }
    ? .systemAndDisplay
    : .systemOnly
```

(This logic predates the multi-bool WakePolicy and currently only considers display sleep. Documented as a future cleanup: "strictest" should `AND` all `allow*` bools — the most-restrictive value across active triggers.)

### Trigger CRUD UI

In **Settings → Triggers**:
- Header with "New Trigger" button.
- List of `TriggerRow` cells: green/gray status dot, name + "N conditions · policy", enabled toggle, edit/delete buttons.
- Editing a trigger opens a sheet with `TriggerEditorView`:
  - Name field
  - Wake-policy picker (legacy enum: System only / System and display)
  - Enabled toggle
  - Conditions list with add/remove (`TriggerCondition` builder per case)

`TriggerDraft` is the editable form-state intermediate; `.toTrigger()` produces the final `Trigger`. `TriggerDraft.isValid` requires non-blank name AND non-empty conditions.

## 10. Persistence

| Datum | Key | Format | Codec |
|---|---|---|---|
| Triggers | `com.pardhu.CafeUp.triggers.v1` | JSON `[Trigger]` | `JSONEncoder` / `JSONDecoder` |
| Icon style | `com.pardhu.CafeUp.menuBarIconStyle.v1` | String raw value | `MenuBarIconStyle.rawValue` |

Storage is `UserDefaults.standard`. There is no migration log file — backward compatibility for `WakePolicy` is handled inline in its `init(from decoder:)`.

### `WakePolicy` migration

```swift
init(from decoder: Decoder) throws {
    // Legacy: a single-value String container with "systemOnly" or "systemAndDisplay".
    if let raw = try? decoder.singleValueContainer().decode(String.self) {
        switch raw {
        case "systemOnly":       self = .systemOnly
        case "systemAndDisplay": self = .systemAndDisplay
        default:                 self = .default
        }
        return
    }
    // New: keyed container with three Bools, each defaulted if missing.
    let c = try decoder.container(keyedBy: CodingKeys.self)
    self.allowDisplaySleep = try c.decodeIfPresent(Bool.self, forKey: .allowDisplaySleep) ?? false
    self.allowSystemSleepWhenLidClosed = try c.decodeIfPresent(Bool.self, forKey: .allowSystemSleepWhenLidClosed) ?? true
    self.allowScreenSaverAfter45Min = try c.decodeIfPresent(Bool.self, forKey: .allowScreenSaverAfter45Min) ?? false
}
```

Triggers persisted before the struct migration continue to load with their previous semantics. Newly-saved triggers use the keyed format.

### Corrupted data

`UserDefaultsTriggerStore.load()` uses `try?` and returns `[]` on any decoding failure. The user effectively starts with no triggers (their stored data remains in defaults — not destroyed — but ignored until manually rewritten).

## 13. App Intents integration

`AppIntentBridge` registers the live `SessionEngine` with `CafeUpIntents` so Shortcuts / focus modes / Siri can drive sessions. Exposed intents start manual sessions (indefinite, timed) and stop the current session. The bridge is set up in `CompositionRoot.makeAppDependencies()`.

## 14. Architecture & module layout

CafeUp follows a strict layered architecture; dependencies point downward.

```
┌─────────────────────────────────────────────────────────────┐
│  App        CafeUpApp · AppDelegate · CompositionRoot       │
│             AppIntentBridge · WindowID · CafeUpIntents      │
├─────────────────────────────────────────────────────────────┤
│  Features   MenuBar · Triggers · Appearance · Settings      │  Views + ViewModels
├─────────────────────────────────────────────────────────────┤
│  Core       SessionEngine · TriggerEngine                   │
```

## 16. Concurrency & thread safety

- `SWIFT_STRICT_CONCURRENCY=complete` is enforced.
- View models, engines, and observers are `@MainActor`-isolated.
- All UI-touching protocol methods are marked `@MainActor`.
- `Scheduler` work closures are `@Sendable () -> Void`; they hop back to the main actor explicitly via `Task { @MainActor in … }` before touching engine state.
- `PowerAssertionToken` is `Sendable` (acquire happens on caller's actor; release is idempotent + locked).
- Notification handlers from `NSWorkspace.notificationCenter` queue on `.main` and use `MainActor.assumeIsolated { … }` to invoke main-actor callbacks.
- IOKit (`IOPMAssertionCreateWithName`, `IOPMAssertionRelease`) is thread-safe per Apple's docs and called without main-actor isolation.

## 19. Known limitations & non-goals

### Limitations
- **Downloads monitor heuristic**: only checks partial-file extensions. A completed download with a sticky temp extension would falsely keep the session alive; an active download to a non-standard extension would not be detected.
- **App lifetime watcher**: doesn't detect *crashes* differently from clean quits — both end the session.
- **Idle observer poll cadence**: 30s default means a session may take up to 30 extra seconds to honor the 45-min screen-saver threshold.
- **`TriggerEngine.strictestPolicy`**: only considers `allowDisplaySleep`. The other two `WakePolicy` bools aren't combined across triggers yet; planned future cleanup.
- **No multiple monitors handling**: `MenuBarIcon` template rendering uses `NSScreen.main?.backingScaleFactor`. Multi-display setups with mixed scale factors may render the custom glyphs at a non-ideal resolution.
- **AppleScript-driven testing of in-panel toggles is not possible**: SwiftUI button labels are not exposed via AXTitle; live verification of those clicks relies on visual snapshots.

### Non-goals
- iCloud sync of triggers / preferences.
- iOS / iPadOS companion.
- Per-display assertions (CafeUp is single-system-wide).
- Custom power profiles beyond the three Bool model.
- Visual themes beyond menu-bar icon selection (the main window respects system appearance).

## 20. Update system

CafeUp ships updates through [Sparkle](https://sparkle-project.org) 2.x. The app **never checks for updates automatically** — every check is initiated by the user via the *Check for Updates…* menu item or the *Check for Updates Now* button in Settings → General → Updates.

### Sparkle Info.plist keys

| Key | Value | Why |
|---|---|---|
| `SUFeedURL` | `https://pthokala.github.io/CafeUp/appcast.xml` | Where Sparkle fetches the update manifest |
| `SUPublicEDKey` | `OAE6wYuiNNZdDgvz4bl+mpQPYZc4nJvXPy08RmB2WwU=` | Public half of the EdDSA keypair; Sparkle uses this to verify the signature on downloaded archives |
| `SUEnableAutomaticChecks` | `false` | Sparkle's background timer is off; no checks run unless the user clicks |
| `SUAllowsAutomaticUpdates` | `false` | The auto-download-and-install opt-in is never offered to the user |

`SparkleUpdaterService.init` also sets `updater.automaticallyChecksForUpdates = false` defensively at runtime, so a stale `UserDefaults` value (from a prior build that flipped it) cannot re-enable background checking.

### EdDSA key management

The signing key is **independent of Apple's Developer ID certificate**. Sparkle uses it to verify that the downloaded archive comes from us (defense against an attacker who compromises the appcast host or the GitHub Release).

**One-time generation** (already done — do not regenerate, doing so invalidates every shipped update):

```bash
xcodebuild -project CafeUp.xcodeproj -scheme CafeUp -resolvePackageDependencies
~/Library/Developer/Xcode/DerivedData/CafeUp-*/SourcePackages/artifacts/sparkle/Sparkle/bin/generate_keys
```

- The **private key** is stored in the macOS login keychain under "https://sparkle-project.org". `sign_update` finds it automatically.
- The **public key** was pasted into `project.yml` as `SUPublicEDKey`.

**Exporting the private key for CI** (one-time):

```bash
~/Library/Developer/Xcode/DerivedData/CafeUp-*/SourcePackages/artifacts/sparkle/Sparkle/bin/generate_keys \
  -x ed_private_key.txt
# Copy the file contents into a GitHub Actions secret named SPARKLE_ED_PRIVATE_KEY
rm ed_private_key.txt
```

### Appcast format

Hosted at `docs/appcast.xml` on `main`, served by GitHub Pages. Sparkle 2 spec. Minimum item:

```xml
<item>
  <title>Version 0.2.1</title>
  <sparkle:version>3</sparkle:version>                 <!-- compared as integer, MUST be monotonic -->
  <sparkle:shortVersionString>0.2.1</sparkle:shortVersionString>
  <sparkle:minimumSystemVersion>14.0</sparkle:minimumSystemVersion>
  <pubDate>Mon, 23 Jun 2026 12:00:00 +0000</pubDate>
  <description><![CDATA[<p>Bug fixes.</p>]]></description>
  <enclosure
    url="https://github.com/pthokala/CafeUp/releases/download/v0.2.1/CafeUp-0.2.1.zip"
    length="2348295"
    type="application/octet-stream"
    sparkle:edSignature="…"/>
</item>
```

### Release process

**Tagged push triggers CI** (`.github/workflows/release.yml`):

1. Workflow verifies all required secrets are present.
2. Imports Developer ID cert into a temp keychain on the macOS-14 runner.
3. `xcodegen generate`, `xcodebuild -resolvePackageDependencies`.
4. `xcodebuild archive` → `xcodebuild -exportArchive` (developer-id method).
5. `codesign --verify --strict` on the .app and the embedded Sparkle.framework.
6. `ditto` → zip → `xcrun notarytool submit --wait` → `xcrun stapler staple` → re-zip.
7. Pipes `SPARKLE_ED_PRIVATE_KEY` via stdin into `sign_update --ed-key-file -` (key never on disk).
8. `gh release create v<version>` with the zip attached.
9. Appends the new `<item>` block to `docs/appcast.xml` and commits back to `main` with `[skip ci]`.

**Required GitHub Actions secrets:**

| Secret | Source |
|---|---|
| `DEV_ID_CERT_P12_BASE64` | `base64 -i DeveloperID.p12` |
| `DEV_ID_CERT_PASSWORD` | Password used when exporting the .p12 |
| `KEYCHAIN_PASSWORD` | Any throwaway password (used only on the runner) |
| `APPLE_API_KEY_ID` | App Store Connect API key id (10 chars) |
| `APPLE_API_KEY_ISSUER` | App Store Connect issuer UUID |
| `APPLE_API_KEY_P8_BASE64` | `base64 -i AuthKey_XXXXXX.p8` |
| `SPARKLE_ED_PRIVATE_KEY` | Contents of `ed_private_key.txt` from `generate_keys -x` |

**Local releases** (no CI):

```bash
scripts/release.sh 0.2.1
```

The script preflights cleanly: aborts if working tree is dirty, if not on `main`, if `project.yml` version mismatches the arg, or if no Developer ID certificate is present. Outputs the `<item>` block ready to paste into `docs/appcast.xml`.

### Version-bump discipline

`CURRENT_PROJECT_VERSION` (build number) **must strictly increase** on every release. Sparkle compares `<sparkle:version>` (which maps to `CURRENT_PROJECT_VERSION`), not the marketing version. Forgetting to bump it is the most common release bug — users see "you're up to date" even though the appcast item is new.

`MARKETING_VERSION` (semver) is what the user sees in About / Settings. Bump per semver discipline.

### Verifying an update flow locally

Without a real Developer ID + Pages site you can still test the UI:

1. Generate a fake higher-version appcast pointing at `file://...` enclosures.
2. Set `SUFeedURL` temporarily to that local file URL (rebuild).
3. Trigger *Check for Updates…* — Sparkle's modal should appear and offer the install (which will fail at signature-verification time, as expected, since you don't have a real EdDSA-signed archive).

### Non-goals

- **Automatic background checks** — explicit user choice; the related `SUEnableAutomaticChecks` and `SUAllowsAutomaticUpdates` are pinned to `false`.
- **Beta channel** (`<sparkle:channel>`) — stable channel only; revisit when there's beta traffic to support.
- **Delta updates** — `generate_appcast`'s binary-delta generation; bundle is currently too small for meaningful bandwidth savings.
- **Signed feeds** (`SURequireSignedFeed`) — extra defense for the appcast XML; worth adding once we're shipping at scale.
- **Custom Sparkle UI** — Sparkle's stock dialog is well-localized and accessible; no reason to roll our own.

## 20.1 Open: appcast hosting

`SUFeedURL` is configured to `https://pthokala.github.io/CafeUp/appcast.xml`, but **GitHub Pages is not currently enabled** on this repo. The first attempt to enable it returns HTTP 422: *"Your current plan does not support GitHub Pages for this repository."* This is because the repo is **private** and GitHub Pages requires a paid plan (Pro/Team/Enterprise) for private repositories.

### Symptoms when unfixed

- *Check for Updates…* in a production build → Sparkle reports a network/feed error after attempting to fetch the URL.
- The local test in § *Verifying an update flow locally* above still works, since it points `SUFeedURL` at a `file://` (or `http://localhost`) URL.
- The release pipeline still **functions** — `xcodebuild archive`, codesigning, notarization, EdDSA signing, GitHub Release creation, and the `docs/appcast.xml` commit all succeed. Only the *delivery* of the feed to end users is blocked.

### Resolution options (in rough order of effort)

1. **Make the repo public.** Standard posture for an open-source menu-bar app; unblocks free Pages immediately. Pre-flip checklist:
   - Audit git history for committed secrets (`git log -p`, `gitleaks detect`, etc.). Repo secrets used by the release workflow live in GitHub Settings, not the tree, so this should come up clean — but verify.
   - Confirm `Sources/Resources/CafeUp.entitlements` and any developer-account references are non-sensitive.
   - Decide on a license (currently none specified) before making the source public.
2. **Upgrade GitHub plan** to Pro / Team / Enterprise for private-repo Pages support.
3. **Host the appcast on a different platform** (Netlify, Vercel, Cloudflare Pages, S3 + CloudFront, your own server). Touches:
   - `project.yml` → `SUFeedURL`
   - `.github/workflows/release.yml` → `RELEASE_URL` interpolation in the appcast-update step
   - Set up CI/CD for the chosen host to deploy `docs/appcast.xml` on every commit to `main`.
   - Decide where the release zip itself is hosted (likely still GitHub Releases — Sparkle's `<enclosure url=…>` and `SUFeedURL` can be on different origins).
4. **Self-hosted update channel** (skip Sparkle's appcast entirely, e.g. a small JSON manifest you maintain manually). Higher maintenance; not recommended.

Until one of the above lands, version `0.3.0` is buildable and signable but won't reach existing `0.2.0` installs via *Check for Updates…*. Users would need to download the GitHub Release zip manually.
